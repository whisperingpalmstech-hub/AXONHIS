2. Introduction
2.1 Purpose
This Functional Requirements Document (FRD) defines the comprehensive functional and technical requirements for the Outpatient Department (OPD) Module within the Hospital Information System (HIS). The OPD module is designed to streamline the complete outpatient care workflow, from patient registration and appointment scheduling to doctor consultation and billing settlement.

2.3 Module Context within Existing HIS
The OPD Module is a core component of the HIS platform and serves as the primary entry point for outpatient care delivery. It integrates seamlessly with multiple HIS modules to ensure continuity of care and efficient hospital operations:
Front Office: Patient registration and appointment management
Billing: Service charges, payment collection, and financial settlement
Nursing: Clinical documentation and vital signs recording
Doctor Desk/EMR: Clinical consultation and prescription management
Pharmacy: Medication orders and dispensing
Laboratory (LIS): Lab test orders and result management
Radiology (RIS): Imaging orders and report management
IPD: Patient admission from OPD
Emergency: Patient transfer and referral
Finance: Revenue tracking and financial reporting
Back Office: Administrative functions and reporting
3. System Architecture and Integration
3.1 Module Architecture
The OPD Module is built on the Advance HIS SAAS platform developed by Axonic Health Pvt Ltd. It leverages cloud-based architecture for scalability, real-time data access, and seamless integration with other HIS modules. The module follows a multi-tier architecture:
Presentation Layer: Web-based user interface accessible via browsers
Application Layer: Business logic and workflow management
Integration Layer: APIs for HIS module communication
Data Layer: Centralized HIS database with patient records
Security Layer: HIS SSO, RBAC

5. Functional Requirements
5.1 Patient Registration
5.1.1 New Patient Registration
The system shall provide functionality to register new patients with the following capabilities: Patient registration can be done by three ways : 1. Manual Registration 2.Quick Registration ( AI assistance / Virtual Secretary) System should support for voice & chat functionality for patient registration by asking questions regarding mandatory fields such as reason to visit, name,  DOB, gender, phone number, e-mail id  3. Registration with ID Scan :By scanning ID (for global use) details should have to reflect on mandatory field boxes 
Capture patient demographics: First name, middle name, last name, date of birth, age, gender
Record contact information: Mobile number, WhatsApp number, email ID, Skype ID
Capture identification details: Identification type (Aadhar, PAN, Passport, etc.), identification number (Refer NFR Document) 
Record patient category: National, General, VIP, Staff, Employee dependent, etc.
Capture additional details: Nationality, marital status, occupation, blood group, religion, race
Record local address: Country, state, city, pin code, address lines
Capture next of kin details: Name, relationship, contact information
Generate unique UHID (Universal Health Identification Number) automatically
Support patient search by UHID, name, mobile number, identification number
Validate mandatory fields before saving
Support document upload (ID proof, photos, etc.)
Generate patient health card
Send registration confirmation via email/SMS
Support employee/dependent registration with employee ID linkage
5.1.2 Patient Search and Update
The system shall allow users to search and update existing patient records:
Search patients by UHID, name, mobile number, date of birth, identification number
Display patient list with key details (UHID, name, DOB, mobile, health card) 
Allow authorized users to update patient demographics and contact information
Prevent duplicate patient registration with validation checks

9. Security Requirements
9.1 Authentication and Authorization
System shall use HIS Single Sign-On (SSO) for user authentication
System shall implement HIS Role-Based Access Control (RBAC)
System shall enforce strong password policies
System shall support multi-factor authentication (MFA) for sensitive operations
System shall automatically log out users after inactivity timeout
System shall prevent concurrent logins from multiple devices (configurable)
System shall maintain user session management
9.2 Data Security
System shall encrypt sensitive data at rest and in transit
System shall mask sensitive patient information (mobile, email, identification numbers)
System shall implement data access controls at field level
System shall prevent SQL injection and XSS attacks
System shall validate all user inputs
System shall implement secure API communication with authentication tokens
9.3 Audit and Compliance
System shall log all user activities with timestamp and user details
System shall maintain audit trail for all data modifications
System shall provide audit reports for compliance monitoring
System shall comply with healthcare data privacy regulations (HIPAA, GDPR, etc.)
System shall support data retention policies as per regulatory requirements
System shall provide data anonymization for research purposes
9.4 Patient Privacy
System shall implement patient consent management
System shall restrict access to patient records based on user roles
System shall provide patient data access logs
System shall support patient data export and deletion (right to be forgotten)
System shall mask patient identity in reports (configurable)
System shall implement break-the-glass access for emergency situations

10. Assumptions and Constraints
10.1 Assumptions
Users have basic computer literacy and training on HIS
Hospital has reliable internet connectivity for cloud-based HIS access
Master data (doctors, specialties, services, tariffs) is configured in HIS
Integration with payment gateway, SMS gateway, and email server is available
Hospital has defined workflows and policies for OPD operations
Users have appropriate devices (computers, tablets) for system access
Hospital has IT support for system administration and troubleshooting
10.2 Constraints
System is dependent on HIS platform availability
Integration with external systems is subject to API availability
Customization is limited to configuration without code changes
Report generation is limited to data available in HIS database
Teleconsultation is dependent on third-party video conferencing integration
Mobile app functionality is limited to web-based responsive design
Offline mode is not supported (requires internet connectivity)
