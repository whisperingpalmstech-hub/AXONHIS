1.3 System Overview
Advance HIS is a Software as a Service (SAAS) platform developed by Axonic Health Pvt Ltd that provides hospitals with network-based access to comprehensive healthcare management tools. The system leverages big data and analytics for improved healthcare delivery, integrating administrative, operational, and clinical management functions. The Procurement Module is a critical component that ensures efficient supply chain management, cost control, and regulatory compliance within the HIS ecosystem.
1.4 Module Context within HIS
The Procurement Module is a core operational module within HIS. It interfaces directly with:
Store & Inventory Module – for stock replenishment and real-time stock updates
Pharmacy Module – for medicine procurement and expiry tracking
OTIMS Module – for surgical consumables and equipment procurement
Vendor Master – for approved vendor management and performance tracking
Fixed Asset Master – for fixed asset creation post-GRN approval
HIS RBAC – for role-based access control and approval authority management

4. Functional Requirements 
For Consumable Items Workflow
4.1 Item Request Management (Consumables Items)
4.1.1 Item Request (FR-IR-001)
Functional Description:
The system shall allow authorised users to create item requests for new items ( Consumable) to be added to the HIS Item Master.
Input Requirements: 
Item Name (Mandatory)  
Manufacturer (Mandatory)
Vendor (Mandatory)  
Expected Price (Mandatory)
Reason for Item Creation (Mandatory) 
Item Type:  Consumable (Mandatory)  
Processing Rules:
User can add multiple items in a single request
All mandatory fields must be filled before saving
System generates unique request number upon saving
Request date is auto-captured as current system date
Requester information is auto-captured from HIS logged-in user
Output/Actions:
Item request saved with status "Pending"
Request routed to approver via HIS workflow engine
Requestor notified via HIS notification engine upon approval/rejection
4.1.2 Item Approval (FR-IR-002)
Functional Description:
The system shall provide an approval interface for authorised personnel to review and approve/reject item requests via the HIS 
Processing Rules:
Only users with approval authority (as per HIS RBAC) can access this function
Approver can view complete item request details
Approver must select items to approve or reject with mandatory remarks on rejection
Status changes to "Approved" or "Rejected" based on action
Approved items move to Item Creation Pending List
4.1.3 Item Creation Pending List (FR-IR-003)
Functional Description:
The system shall display all approved item requests pending creation in the HIS Item Master.
Processing Rules:
Display only approved item requests
System auto-routes to Item Master if Consumable
User completes additional mandatory fields: Item Category, Product Category, Formulation Type, Brand Name, Generic Name, Batch Required, Is Consumable, UOM, Purchase Type, Stock Type, Dispensing Type, IP/OP/Store UOM
Upon saving, item is created in HIMS Item Master 
Item removed from pending listItem Master :  Check for Item type ( consumable ), Item category, Product category, formulation type, item code, product code, Brand Name/Item Name, pharmacological classification, manufacturer, purchase tax, sale tax, Check for flags -> Item Reusable, Is Vaccine, Batch Required, High Risk, Is Kit, Is Infusion, Hospital Formulary, Is Consumable, Is Injection, Dose calculation applicable, Is OTC, Is Emergency Drug, Is LASACheck for Purchase type UOM unit, stock type UOM Unit, IP UOM, OP UOMProvision to add items -> After creating items -> purchase request create against that item

4.2.1 Purchase Request Creation (FR-PR-001)
Functional Description: 
The system shall allow users to create purchase requests for items that need to be procured, either as Direct Requests or Against Indent. 
Input Requirements: 
Request Type: Consumable (Mandatory) 
PR Type: Direct Request or Against Indent (Mandatory)
Requesting Store (Mandatory)
Delivery Store (Mandatory)
Priority Level (Mandatory)
Item Selection from HIMS Item Master (Mandatory)
UOM, UOM Unit (Mandatory) 
Purchase Quantity (Mandatory)
Specifications (Optional)
Remarks (Optional)
Processing Rules:
User clicks "Add Item" to open item selection from HIMS Item Master
System displays current stock and pending quantity for selected items
System validates that all mandatory fields are filled
Upon saving, PR is created with status "Pending" and unique PR number
System validates PR quantity against budget limits configured in HIS Finance Module
Output/Actions:
PR saved and routed to designated approver via HIS 
PR status displayed at all times:  Pending Approval / Approved 
4.2.2 Purchase Request Approval (FR-PR-002)
Functional Description:
The system shall provide a multi-level approval workflow for purchase requests via the HIS 
Processing Rules:
Only authorised approvers (as per HIS RBAC) can access this function
Approver views PR details including items, quantities, and specifications
Approver can Accept or Return the PR for correction with mandatory remarks
Approved quantity can be modified by approver
Upon approval, PR status changes to "Approved" and linked to Quotation Enquiry

The system shall allow users to create quotation enquiries Against PR or as Direct Enquiry, to be sent to potential suppliers.
Input Requirements:

Type: Consumable (Mandatory)
Against: PR or Direct (Mandatory)

UOM, UOM Unit (Mandatory)

Status set to "New/Pending" until sent to vendors
4.3.2 Enquiry Sent to Vendor (FR-QT-002)
Functional Description:
The system shall facilitate sending quotation requests to selected vendors from the HIS Vendor Master.
Processing Rules:
User clicks "Add Enquiry" and selects store, type, and vendor(s) from HIS Vendor Master
User clicks "Get QR" to retrieve quotation requests
System generates Vendor Quotation Request (VQR) number for each vendor
VQR sent to vendor with due date
Status updated to "Sent to Vendor"
4.3.3 Quotation Fill In (FR-QT-003)
Functional Description:
The system shall provide an interface for entering vendor quotation details received from suppliers.
Input Requirements:
Store (Mandatory)
Type (Mandatory)
Vendor (Mandatory)
Reference Number (Mandatory)
VQR Selection (Mandatory)
Vendor COP (Mandatory)
Accessories Details (Optional)
Support Terms, Warranty Period, Technical Specifications (Optional)
Remarks (Optional)
Processing Rules:
User clicks "Get VQR" to retrieve vendor quotation requests
User enters vendor pricing and terms for each item
User can select All and click SAVE to save all vendor quotations
Quotation status changes to "Filled" and moves to vendor negotiation stage
4.3.4 Vendor Negotiation (FR-QT-004)
Functional Description:
The system shall provide an interface for negotiating terms with vendors based on received quotations, with side-by-side comparison of user requirements vs vendor offered details.
Processing Rules:
System displays comparison: COP, Warranty Period, Support Terms, Technical Specifications, Payment Terms
User can enter negotiated details for each term
User selects action: Accept, Reject, or On Hold for each line
For acceptance, user clicks "Send For Vendor Selection"
Status changes to "Completed" upon vendor selection
4.3.5 Vendor Selection (FR-QT-005)
Functional Description:
The system shall allow final vendor selection after the negotiation process, with a vendor comparison view.
Processing Rules:
User reviews negotiated quotations using Info icon for vendor comparison matrix
User selects "Accept" for chosen vendor and clicks "Send For PO Creation"
Status changes to "Closed"
Selected quotation routed to Purchase Order creation workflow
4.4 Purchase Order (PO) Management
4.4.1 Purchase Order Creation (FR-PO-001)
Functional Description:
The system shall enable creation of purchase orders based on approved quotations (Against VQR), direct PO, or Against PR.
Input Requirements:
Delivery Store (Mandatory)
Requesting Store (Mandatory)
Vendor (Mandatory)
PO Type: Consumable or Fixed Asset (Mandatory)
Source: Against Approved VQR / Direct PO / Against PR (Mandatory)
Item Details, Quantity, UOM, COP, MRP (Mandatory)
Discount, Other Charges (Optional)
Tax Details (Mandatory)
Processing Rules:
User selects source and adds items via ADD QR / ADD PR functionality
System calculates: Total Amount, Total Discount, Total Tax Amount, Total Other Charges, Round Off, Net Amount
Upon saving, PO created with unique PO number and status "Pending Review"
4.4.2 PO Review (FR-PO-002)
Functional Description:
The system shall provide an interface for reviewing and modifying purchase orders before final approval.
Processing Rules:
Reviewer can modify: Quantity, COP, MRP, Discount, Other Charges
Reviewer can print or cancel PO
Upon saving changes, PO moves to approval stage with status "Under Review"
4.4.3 PO Approval (FR-PO-003)
Functional Description:
The system shall provide multi-level approval workflow for purchase orders via the HIMS workflow engine.
Processing Rules:
Only authorised approvers (as per HIMS RBAC) can access this function
Approver can Accept or Reject PO with mandatory remarks on rejection
Upon acceptance, PO status changes to "Approved"
System generates PO document with: PO Number and Date, Vendor Details, Item Details with HSN Code, Quantities, Pricing, Discount and GST details, Net Amount, Terms & Conditions, Authorised Signatures
Approved PO linked to GRN module and ready for GRN processing
4.5 Goods Receipt Note (GRN) Management
4.5.1 GRN Creation (FR-GR-001)
Functional Description:
The system shall enable creation of Goods Receipt Notes when materials are received from vendors, either Against PO or as Direct GRN.
Input Requirements:
Requesting Store (Mandatory)
Delivering Store (Mandatory)
Type: Consumable or Fixed Asset (Mandatory)
GRN Type: Against PO or Direct GRN (Mandatory)
Vendor (Mandatory)
Vendor Invoice Number (Mandatory)
Inward Number (Mandatory)
DC (Delivery Challan) Number (Mandatory)
PO Selection (for Against PO GRN) (Mandatory)
Batch Number (Mandatory)
Expiry Date (Mandatory for pharmaceuticals)
Received Quantity (Mandatory)
Bonus Quantity (Optional)
COP, MRP, Tax Details (Mandatory)
Discount, Specifications, Remarks (Optional)
Processing Rules:
User selects GRN type and adds PO items via ADD PO functionality
System validates received quantity against PO quantity
System calculates: Total Amount, Total Discount, Total Tax Amount, Credit Note (if applicable), Round Off, Net Amount
Upon saving, GRN created with unique GRN number and status "Pending Approval"
4.5.2 GRN Approval (FR-GR-002)
Functional Description:
The system shall provide approval workflow for Goods Receipt Notes with barcode generation and physical inspection support.
Processing Rules:
Approver can view, print, edit, and approve or reject GRN
System supports Barcode Print and Edit functionality from GRN Approval screen
System supports optional Physical Inspection Flag on GRN Approval
Approver can accept or reject items with mandatory reason for rejection (Cancel, In Stock, Wrong Item)
Upon approval, GRN status changes to "Approved"
System generates final GRN receipt with all details including Tax Breakdown (CGST, SGST)
4.5.3 GRN Settlement (FR-GR-003)
Functional Description:
The system shall facilitate payment settlement for approved GRNs with automatic stock and financial updates in HIS.
Input Requirements:
GRN Selection (Mandatory)
Payment Mode: Cash, Cheque, or E-Transfer (Mandatory)
Payment Amount (Mandatory)
Bank Details (for Cheque/E-Transfer) (Mandatory)
Transaction Number (for E-Transfer) (Mandatory)
Cheque Number (for Cheque) (Mandatory)
Processing Rules:
User marks approval checkbox ON and clicks Settle
System automatically updates stock levels in HIMS Store & Inventory Module in real time
System automatically posts financial entries to HIMS Finance Module (invoice matching and ledger update)
Settled GRN moves to Settled List Tab
Audit trail entry created in HIS centralised audit trail
4.5.4 Physical Inspection (FR-GR-004)
Functional Description:
The system shall provide an interface for physical inspection of received goods.
Processing Rules:
Quality inspector accesses closed GRNs
Inspector verifies: Damage, Quality issues, Quantity discrepancies, Expiry dates
Inspector records inspection results
If issues found, inspector can initiate return process
Inspection status updated in HIMS

System validates payment information
Upon settlement: Payment recorded, Work order moved to settled list, Vendor account updated, Status changes to "Settled"
User can print work order receipt

7. Security Requirements
7.1 Authentication and Authorisation
All users shall authenticate via HIMS Single Sign-On (SSO)
Access to procurement functions shall be governed by HIMS RBAC with role and department-level control
Approval authority limits (e.g., PO value thresholds) shall be configurable per role in HIMS RBAC
Sensitive procurement data (vendor pricing, PO values) shall be accessible only to authorised roles
System shall enforce session timeout as per HIMS security policy
Multi-factor authentication support (optional, configurable)
7.2 Data Security
All procurement data shall be encrypted in transit (TLS 1.2+) and at rest
Vendor pricing and quotation data shall be treated as confidential and access-restricted
System shall prevent unauthorised modification of approved POs and settled GRNs
Amend functionality for approved records shall be restricted to authorised roles only
Secure API communication using HTTPS for all HIMS integrations
Regular security audits and vulnerability assessments
7.3 Audit and Compliance
System shall log all procurement actions (create, edit, approve, reject, settle) with user ID, timestamp, and IP address
Audit logs shall be immutable and accessible only to authorised auditors
System shall support regulatory compliance reporting for procurement activities
All procurement records shall be retained for a minimum of 7 years as per hospital policy
Compliance with hospital procurement policies and regulatory standards for medical equipment procurement
GST compliance for all financial transactions

8. Assumptions and Constraints
8.1 Assumptions
The HIS platform (Advance HIS by Axonic Health Pvt Ltd) is deployed and operational as the parent system
HIS RBAC is configured with all required procurement roles and approval hierarchies prior to go-live
Vendor Master, Item Master, and Fixed Asset Master are pre-populated with baseline data before procurement module activation
The Finance Module is integrated and configured for budget validation and financial posting
The Store & Inventory Module is integrated for real-time stock updates upon GRN settlement
All users have valid HIS SSO credentials and appropriate role assignments
Network connectivity is available for all procurement users to access the HIS cloud platform
Hospital procurement policies (approval limits, vendor selection criteria) are documented and configured in HIS
Vendors can provide quotations in the required format
8.2 Constraints
The Procurement Module must operate within the HIS SaaS architecture and cannot be deployed as a standalone system
All integrations must use HIS-approved APIs; direct database access is not permitted
Approval workflows are configurable but must follow the HIS workflow engine framework
Report formats must conform to HIS standard reporting templates
Data retention policies are governed by hospital regulatory requirements and HIS platform capabilities
System performance depends on internet bandwidth
Mobile access may have limited functionality
Third-party integration dependent on API availability
