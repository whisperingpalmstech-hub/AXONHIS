2. Introduction
2.1 Purpose
This Functional Requirements Document (FRD) defines the functional for the Store and Inventory Management module of the Hospital Information System (HIS). The Store and Inventory module is a critical component that manages the complete lifecycle of hospital inventory including procurement, storage, distribution, tracking, consumption, stock adjustments, and disposal. This document serves as a comprehensive guide for developers, testers, and stakeholders involved in the implementation and maintenance of the module.

1. Store creation & min/max configuration
Min & max value definition
Each product in each store (main store or sub-store) is configured with a minimum level and a maximum level. These thresholds drive all automated alerts and indent generation. Values are set per store per item — ICU, pharmacy, ward stores can each have distinct min/max thresholds for the same product.
Auto alert / indent visibility on reaching min level (Automated)
When stock of any item touches or drops below the configured minimum, the system automatically generates an alert and a pending indent. This indent is immediately visible to the store manager on their dashboard — no manual scan needed.
            2. VED classification & automatic indent dispatch
VED-based indent routing
                     Products are classified as Vital (V), Essential (E), or Desirable (D).
                      Vital — indent auto-dispatched immediately
                       Essential — indent raised, manager review
                        Desirable — standard indent queue
For Vital items: as soon as the quantity hits the minimum level, the purchase indent is automatically sent without any manual trigger. No intervention required.
             3. Order restriction & vital auto-indent logic
Order restriction rule
Orders (indents) can be restricted to ensure they are only raised when stock is at or below the defined minimum. Prevents over-ordering and maintains max-level discipline.
Vital indent — fully automatic dispatch
Vital category items require no manual action. The moment stock touches the min value, the indent is auto-generated and auto-dispatched to the relevant store or purchase department.
              4. All indents visible on a single screen
The store manager sees all pending, auto-generated, and approved indents on one consolidated screen — filterable by store, VED class, status, and date.Example: 
Paracetamol 500mg    Vital           Qty: 200Auto-dispatched
IV Fluid NS    500ml     Essential    Qty: 50Pending approval
Surgical Gloves L          Desirable   Qty: 100In queue
               5. Override indent & auto approval
Override facility for store manager (Configurable)
The store manager is given the ability to override a system-generated indent — change quantity, defer, or escalate — within defined authorization bounds. An override is logged with reason and user ID for audit.
Auto approval — single-level only
The system supports auto-approval for indents within pre-configured quantity and value bands (e.g., quantity between 50–100 units). These require only one level of approval — no multi-tier routing. No manual intervention is required for routine, within-range indents.

7. Dynamic max value change (ICU / emergency / poisoning)

Provision to change max value on-the-flag (Emergency use)
In exceptional clinical scenarios — such as ICU surge, mass poisoning, or outbreak — the system allows authorised users to temporarily increase the max level of a product. This bypasses the standard max-limit restriction so that emergency indents for higher quantities are permitted. Change is logged with timestamp, reason, and authoriser for compliance. 8. Check date & quantity band (50–100 range, no manual step)
System-driven check date assignment (Automated)
The system auto-assigns a check date based on consumption rate and min/max configuration. Quantity bands — e.g., 50 to 100 units — are pre-defined per product. When an indent falls within the configured band, it is processed automatically with no manual entry required. Only one approval level is needed before dispatch.

Scenario: Injection Vials — Main Store to 10 Sub-Stores
The hospital has 10 sub-stores (wards, ICU, OT, OPD, etc.), each configured with a minimum and maximum level for a particular injection vial.
When stock in each sub-store touches its minimum, the HIS auto-generates an indent. Each sub-store requires 10 vials to reach its max level. With 10 sub-stores, the total consolidated indent demand coming to the main store is 100 vials.
At the same time, the main store's own stock is sitting at 50 vials — below its configured minimum — which triggers a purchase request to the supplier. The supplier is asked to supply 100 vials to replenish the main store.
The HIS now applies the following logic automatically:
Main store stock on hand: 50 vials
Incoming supply from supplier: 100 vials
Total available at main store: 150 vials
Sub-store indent demand: 100 vials
After fulfilling all sub-stores: main store retains 50 vials as its own buffer stock
So the purchase request quantity of 100 is calculated by the system to satisfy both the sub-store indents and restore the main store back to its minimum level — the system ensures 50 (existing) + 100 (supplied) = 150, of which 100 goes out to sub-stores and 50 stays in the main store. No manual calculation is required. The HIS auto-computes the purchase request quantity to achieve the complete desired stock position across the entire hierarchy in a single procurement cycle.

2.3 Module Context within HIS
The Store and Inventory module is an integral part of the Hospital Information System (HIS) and serves as the central hub for managing all hospital inventory across departments. It interfaces with:
Procurement Module: Receives purchased items through GRN and manages stock replenishment
Pharmacy Module: Supplies pharmaceutical items and tracks pharmacy inventory
OTIMS (Operation Theatre): Provides surgical supplies and tracks OT consumption
IPD/OPD Modules: Supplies consumables and tracks patient-related consumption
Laboratory Module: Provides lab consumables and reagents
Billing Module: Integrates consumption data for patient billing
Finance Module: Provides inventory valuation and cost tracking
CSSD, Housekeeping, IT Department: Supplies department-specific items
3. System Architecture and Integration

The Store and Inventory module is built as a core component of the HIS platform with the following architectural layers:
Presentation Layer: Web-based responsive UI accessible through HIS portal
Business Logic Layer: Inventory management rules, workflow engine, approval mechanisms
Data Access Layer: Centralized HIS database with inventory-specific schemas
Integration Layer: APIs for seamless integration with other HIS modules
Reporting Layer: Real-time dashboards and comprehensive reports

The Store and Inventory module follows these data flow patterns:

Stock Distribution: Store and Inventory → Departments (Material issue workflow)
Consumption Tracking: Departments → Store and Inventory → Billing/Finance
Stock Adjustments: Store and Inventory → Finance (Valuation updates)
Disposal: Store and Inventory → Finance/Audit (Scrap and disposal records)
Reporting: Store and Inventory → Management Dashboard (Real-time analytics)

6. Process Workflows
6.1 Store Indent Workflow
1. Department creates store indent
2. System validates indent and generates indent number
3. Indent routed to Department Head for approval
4. Department Head approves/rejects indent
5. Approved indent appears in Store issue worklist
6. Store checks stock availability
7. Store issues materials based on FIFO/FEFO
8. System updates stock levels
9. Receiving department accepts/rejects materials
10. System updates department inventory
11. Indent marked as completed 
6.2 Material Issue Workflow
1. Store selects approved indent
2. System displays available stock with batch details
3. Store selects batch based on FIFO/FEFO logic
4. Store verifies expiry date
5. Store enters issued quantity and carrier details
6. System prints material issue slip
7. System updates stock levels
8. Notification sent to receiving department
9. Receiving department verifies received items
10. Receiving department accepts/rejects in system
11. System updates department inventory
12. Issue marked as completed
6.3 Material Return Workflow
1. Department creates material return request
2. System validates return and generates return number
3. Department prints return slip
4. Materials physically returned to store
5. Store verifies returned items  
6. Store inspects item condition 
7. Store accepts/rejects return in system
8. System updates store stock levels 
9. System updates department inventory  
10. Notification sent to returning department
11. Return marked as completed 
6.4 Stock Adjustment Workflow
1. Store identifies need for stock adjustment
2. Store creates adjustment request with reason
3. System validates adjustment details 
4. Adjustment routed to Store In-charge for approval
5. Store In-charge reviews adjustment and supporting documents
6. Store In-charge approves/rejects adjustment
7. System updates stock levels upon approval
8. System updates inventory valuation
9. Notification sent to Finance module
10. Adjustment marked as completed
6.5 Scrap and Disposal Workflow
1. Store creates scrap request for unusable items
2. System validates scrap details
3. Scrap request routed to Store In-charge for approval
4. Store In-charge reviews and approves/rejects
5. Approved items moved to disposal worklist
6. Disposal team executes disposal as per approved method
7. Disposal details recorded in system
8. Witness assigned for verification
9. Witness verifies disposal execution
10. Witness approves disposal in system
11. System updates stock levels
12. System updates inventory valuation
13. Disposal certificate generated
14. Notification sent to Finance and Audit
15. Disposal marked as completed
